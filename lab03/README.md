# Документация для третьей лабораторной работы по ООАП

# Тема лабораторной: Поведенческие паттерны

---

Предметная область: Система расходов

1. Проблема: В системе согласования корпоративных расходов разные уровни руководства имеют разные финансовые лимиты. Жёсткая привязка формы заявки к конкретным сотрудникам через вложенные `if/else` нарушает принцип единственной ответственности и делает архитектуру хрупкой: добавление новой должности или изменение лимитов требует переписывания центральной логики маршрутизации, а клиентский код вынужден знать всю внутреннюю иерархию компании.
2. Решение: Мы выносим логику утверждения в цепочку независимых объектов (`Handler` и наследники `ManagerHandler`, `DirectorHandler`, `VicePresidentHandler`). Каждый обработчик хранит ссылку на следующего в поле `_next_handler` и отвечает только за проверку своего лимита. Если сумма превышает допустимую, запрос автоматически делегируется дальше через `_pass_to_next()`. Клиент (`ApprovalApp`) взаимодействует только с первым звеном цепи, полностью абстрагируясь от того, сколько уровней и кто именно обработает заявку, что позволяет динамически менять порядок и состав согласующих лиц без изменения кода отправки.

4. Вывод:
Паттерн Chain of Responsibility успешно решает задачу децентрализации логики согласования. Вместо монолитного метода с множеством условий мы получаем гибкую систему, где каждый обработчик инкапсулирует знания только о своём лимите.
